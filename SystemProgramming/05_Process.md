## 메모리 구조

- 전역변수 혹은 static 변수의 할당을 위해 존재하는 `Data 영역`
- 지역 변수의 할당과 함수 호출 시 전달되는 인자값들의 저장을 위해 존재하는 `Stack 영역`
- 동적 할당을 위해 존재하는 `Heap 영역`
- 실행파일을 구성하는 명령어들이 올라가는 메모리 영역을 가리켜 `Code 영역`
- 레지스터들의 상태까지도 프로세스의 일부로 포함시켜 말할 수 있다.
    - 가령 예를 들어 swift.exe가 실행 되면 CPU를 구성하는 레지스터들은 swift.exe의 실행을 위해 필요한 데이터들로 채워지게 됩니다!
    

## Process의 스케줄링(Scheduling)

- 스케줄링의 기본 원리
    - 프로세스의 CPU 할당 순서 및 방법을 결정짓는 일을 가리켜 스케줄링, 이때 사용되는 알고리즘을 스케줄링 알고리즘(Scheduling Algorithm)이라 한다.
- 스케줄링 방식을 두 가지로 생각해볼 때,
    - A, B, C를 순차적으로 실행시킨다. A 완료 후 B, B 완료 후 C
    - A, B, C를 동시 실행, I/O 같이 오래걸리는 작업이라면 다른 Process가 실행되도록 한다.
    - 두 번째 방법이 CPU를 쉬지 않고 일하게 할 수 있다.

## 프로세스의 상태 변화

- Start: 프로세스의 생성, 프로그램이 실행 됨.
- Ready: CPU에 의해 실행되기를 기다리는 상태를 의미한다.
- Running: 실행중인 프로세스
- 프로세스의 중요도에 따라 우선순위(Priority)가 매겨지는데, Running 도중 더 높은 우선순위의 프로세스가 생성되면 Running → Ready로 상태 변화 한다.
- blocked: 스케줄러에 의해서 선택될 수 없는 상태를 의미한다. `Ready의 상태는 스케줄러에 의해 선택되어 지금 당장이라도 실행 가능한 상태이고, Blocked 상태는 스케줄러에 의해서 선택될 수 없는 상태이다.

## Context Switching

- 정의: 하나의 프로세스가 실행되고 있는 상태에서, 인터럽트 요청에 의해 다음 우선순위 프로세스가 실행되어야 할 때 기존의 프로세스 상태 또는 레지스터 값(Context)를 저장하고 CPU가 다음 프로세스를 실행하도록 다음 프로세스의 상태 또는 레지스터 값(Context)를 교체하는 작업.
- Conetxt란? 프로세스를 실행하기 위해 해당 프로세스의 정보들이다.
